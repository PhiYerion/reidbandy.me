IRA

A high precision (1024 bit) multi-stage rocket program & calculator.

Multi-precision Floating Point Arithmetic


We use gnu mpfr (The Multiple Precision Floating-Point Reliable Library) to achieve highly customize-able, accurate, and precise calculations. Any normal floating point numbers and calculation implementations are simply no where near usable.

Using mpfr creates a variety of issues that will have to be solved:

1. Each class that uses an mpfr type requires custom copy and move operations as the mpfr type is a class that contains a pointer to memory which is handled in an incompatible way to the default move and copy operations.

2. Mpfr has a cache that has to be cleared at an optimal time.

3. Mpfr variables have to be explicitly deleted

4. Parsing a conventional floating point into an mpfr type would immediately reduce the accuracy.

5. Parsing any input into an mpfr type and back is hard to read and takes a relatively large amount of code

Memory Management, Ownership, and Modularity


Ship, Stage, and Engine:
The space ship is broken down in a straight forward manner (a ship have stages unique to the ship, stages have an engine not unique to the stage.). Usually, there isn't a need for engines or stages to know what ships/stages they belong to and thus isn't implemented. Due to the Ship Wrapper taking care of conversions, the Ship, Stage, and Engine classes only deal in mpfr_t variables unless required otherwise. The Ship class is responsible for all calculations and has ownership of stages but does not have any independent variables (besides stage count, if you count that). The Stage class only points to it's engine (without ownership) and only contains independent variables.

Ship Wrapper:
Due to the issues with dealing with mpfr types and to create a consistent interface to the program independent of other changes, a wrapper is implemented. This wrapper converts strings into types of mpfr_t. Strings are used as they have arbitrary precision and customize-able base system. This is the only realistic way to achieve true conversion.

Handler:
Due to the issues with dealing with mpfr memory management and handling the set of engines and ships, a "handler" was implemented. This handler can be used to search for ships or engines by name via a hash map. Upon destruction, the handler properly destroys engines and ships along with clearing the cache.

Destruction chain:
Either the main object will exit or specifically delete the Handler object. The Handler object first deletes the ShipWrapper(s), which implicitly deletes it's Ship child, which clears it's mpfr_t variables and deletes all stages, which clears all it's mpfr_t variables. Finally, the Handler deletes all Engines and clears the mpfr cache.

Testing & Verification

Rigorous testing has been implemented to ensure that accurate and precise values are implemented.
To make sure that build methods and the program itself works on most Linux distributions, there are docker files for Fedora, Arch, Debian, and Alpine. This is especially important due to to linkage and dependencies, and this has improved build cross-compatibility. 

Testing uses the Catch2 framework. There are three test cases:

Accuracy - Validates that calculations and setting variables produces accurate numbers. Outputs are compared against 1. numbers generated and cross-checked from multiple reliable and high precision calculators and 2. a previous known 'good run'.

Runtime Errors - Does a series of unpredictable (random) actions that could trip up one of the many systems, memory handling, pointers, etc.

"Crazy" - Generates 1000 unique ships with 10-30 stages with random values from 0.0-1,000,000,000.0. For each stage in each ship, there is a
1/256 to 255/256 chance of a modification for each respective value in each stage. The outputs are compared against a relatively simplistic Delta-V calculation that is known to be good (albeit with less precision).
